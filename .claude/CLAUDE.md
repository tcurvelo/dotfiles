# Personal Guidelines

## Communication

- Be clear and concise. Default to brief answers; I will ask when I need more detail.
- Lead with the recommendation or answer; give reasoning only when asked or when the call is non-obvious.
- Before sending, cut anything not needed to answer.
- Do not use em dashes. Prefer parentheses, commas, and semicolons where they fit.

## Code style

- Comments: keep concise. Prefer one-liners; push depth into docs rather than inline blocks. Match the comment density of the surrounding code.
- Same em dash rule applies to prose and comments.
- Question whether new code is necessary. Prefer removing over adding; flag speculative code and abstractions proactively.
- Before writing non-trivial code, find the nearest existing sibling and match its conventions; say which one.

## Git

- Commit messages: concise, one-liner preferred.
- Do not add agents as co-authors unless explicitly requested.
